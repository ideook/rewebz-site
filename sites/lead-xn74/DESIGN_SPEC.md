## 1) 브랜드 포지션/톤

### 브랜드 포지션
**“서울 팀의 운영 속도를 높여주는 실시간 알림 파트너”**

- 단순 메시지 전송 도구가 아니라, **문의 누락·응답 지연·운영 리스크를 줄이는 서비스**로 포지셔닝
- 핵심은 기능 나열보다 **실무 성과(빠른 대응, 놓침 감소, 업무 안정화)** 전달
- 랜딩의 역할은 “좋아 보임”이 아니라 **도입 문의 행동 유도**

### 톤 앤 매너
- **톤**: 명확하고 신뢰감 있는 실무형 톤
- **카피 원칙**
- 추상어 최소화, 결과 중심 문장 사용
- 기술 용어는 짧게 풀어 설명
- 과장 대신 기준·프로세스 기반 표현
- **지양**
- “무조건 성공”, “완벽 보장” 식 단정형 카피
- 불안만 자극하는 과도한 문제 제기
- 기능명만 나열한 설명식 문장

## 2) 핵심 사용자와 시나리오

### 핵심 사용자
1. **운영/CS 담당자**
- 니즈: 문의/장애/결제 관련 이벤트 즉시 확인
2. **마케팅·세일즈 담당자**
- 니즈: 리드 발생 알림을 빠르게 받아 후속 대응 속도 향상
3. **팀 리더/의사결정자**
- 니즈: 도입 난이도, 안정성, 유지관리 부담 확인 후 판단

### 사용자 시나리오
- **시나리오 A (문제 인식 유입)**
검색 유입 → “알림 누락 문제” 공감 → 해결 방식 확인 → 문의 제출
- **시나리오 B (비교 검토 유입)**
타 솔루션과 비교 → 차별점/신뢰 요소 확인 → 상담 요청
- **시나리오 C (모바일 즉시 행동)**
핵심 가치 확인 → CTA 클릭 → 최소 입력으로 문의 완료

## 3) 정보구조(IA)와 페이지 섹션 순서

### IA 설계 원칙
- 흐름: **문제 인지 → 해결 이해 → 신뢰 확보 → 문의 전환**
- 각 섹션은 사용자의 질문 하나에 답하는 구조로 설계
- CTA는 상단/중단/하단에 반복 배치

### 페이지 섹션 순서
1. **헤더**
- 브랜드명, 핵심 가치 한 줄, 문의 CTA
2. **Hero**
- 문제 해결형 헤드라인 + 보조 설명 + 1차 CTA
3. **문제-해결 섹션**
- 알림 누락의 비용 제시 → 텔레그램 알림 구조 설명
4. **핵심 기능/활용 섹션**
- 실시간 전달, 조건 분기, 팀 공유, 운영 자동화
5. **신뢰 증빙 섹션**
- 운영 기준, 지원 정책, 적용 사례 요약
6. **FAQ 섹션**
- 연동 방식, 보안, 비용, 구축 기간
7. **최종 문의 섹션**
- 간편 폼 + 예상 응답 시간 안내
8. **푸터**
- 기본 정보, 연락 채널, 정책 링크

## 6) 차별화 포인트(기존 흔한 디자인과 어떻게 다를지)

### 흔한 랜딩의 한계
- 기능 목록 중심으로 끝나 도입 판단 근거 부족
- 템플릿형 비주얼로 브랜드 신뢰도 약함
- CTA 반복만 많고 문의 설득 흐름이 약함
- 폼 입력 장벽으로 전환 손실 발생

### 차별화 전략
1. **운영 리스크 관점 문제 정의**
- 알림 누락이 만드는 실제 손실을 먼저 제시
2. **결과 중심 메시지 구조**
- 기능보다 대응 속도/누락 방지/운영 안정 강조
3. **신뢰형 콘텐츠 설계**
- 보안/지원/도입 절차를 명확히 안내
4. **모바일 전환 최적화**
- 고정형 CTA + 최소 입력 폼
5. **맥락 기반 시각 톤**
- 텔레그램 연상 컬러를 절제해 전문성 유지
